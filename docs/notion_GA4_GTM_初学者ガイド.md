# GA4 / GTM 設定 初学者ガイド

このガイドは「計測（けいそく＝サイトでのユーザー行動をデータとして記録すること）」を初めて学ぶ人向けに、**GA4** と **GTM** の役割・設定の流れ・このデモサイトでの確認方法を、ひとつずつ図でたどれるようにまとめたものです。

各章は **原則（考え方） → 図解（全体像） → 具体手順 → このデモとの接続** の順で並んでいます。専門用語は初めて出てきたところに短い注釈を付けています。

---

## 0. この章で使う言葉（最初に3語だけ）

| 用語 | 読み | ひとことで言うと |
|---|---|---|
| GA4 | ジーエーフォー | Google Analytics 4。アクセス・行動データを**ためて分析する道具**（受け皿） |
| GTM | ジーティーエム | Google Tag Manager。どのタグをいつ送るかを管理する**司令塔** |
| dataLayer | データレイヤー | サイトとGTMが値をやり取りする**伝言板**（JavaScriptの配列） |

> 補足: 「タグ」とは、計測やマーケのために外部サービス（GA4・広告など）へデータを送る小さなコードのことです。

## 1. 全体像 ── サイト → dataLayer → GTM → GA4

まず「データがどこを通ってGA4に届くのか」を1枚で押さえます。

### 原則

サイトのコードは、GA4に直接データを送りません。間に **dataLayer（伝言板）** と **GTM（司令塔）** を挟みます。サイトは「何が起きたか」を伝言板に書くだけ。GTMがそれを読んで、GA4が分かる形に変換して送ります。

### 図解：4者の関係

```text
┌──────────────┐   push     ┌──────────────┐   読み取り   ┌──────────────┐  変換・送信  ┌──────────────┐
│   サイト     │ ─────────▶ │  dataLayer   │ ──────────▶ │     GTM      │ ──────────▶ │     GA4      │
│ (HTML/JS)    │  値を書く  │  (伝言板)    │             │  (司令塔)    │             │  (受け皿)    │
│              │            │ JS の配列    │             │ いつ・何を   │             │ 分析・レポート│
│ 例:カート追加│            │ event/items  │             │ 送るか判断   │             │              │
└──────────────┘            └──────────────┘             └──────────────┘             └──────────────┘
       │                            │                            │                            │
   「起きた事を                「値の置き場」               「ルールで                  「ためて
    書くだけ」                                              振り分け」                   見える化」
```

### たとえで覚える

| 登場人物 | たとえ | 仕事 |
|---|---|---|
| dataLayer | サイトとGTMの **伝言板** | 「カートに追加された」「この商品を見た」を書き残す |
| GTM | タグの **司令塔** | 伝言板を見て「どのタグを・いつ・何を送るか」を指示する |
| GA4 | 分析の **受け皿** | 送られてきたデータをためて、グラフ・表で見せる |

### このデモとの接続

このデモサイト（UNILO）では、サイトが値を書く部分が `assets/js/datalayer.js` です。ファイル冒頭にもこの流れが書かれています。

```text
[サイトのコード] --push--> [dataLayer] --読み取り--> [GTM] --変換--> [GA4]
```

サイトのボタン操作 → `window.DL.addToCart(...)` などが呼ばれる → `window.dataLayer.push(...)` で伝言板に書かれる、という順で動きます。

## 2. GA4とは / GTMとは ── 役割の違い

### 原則

GA4とGTMは **別の道具** です。よく混同されますが、片方が「ためる・見る」係、もう片方が「送る・管理する」係です。

### 図解：役割分担

```text
┌─────────────────────────────┐        ┌─────────────────────────────┐
│            GTM              │ 送る   │            GA4              │
│  「タグの司令塔」          │ ─────▶ │  「分析の受け皿」          │
│                            │        │                            │
│ ・どのタグを置くか管理     │        │ ・データを保存             │
│ ・いつ送るか(トリガー)     │        │ ・レポート/探索で分析      │
│ ・何を送るか(変数で取得)   │        │ ・キーイベント(CV)集計     │
│ ・コードを触らず変更可     │        │ ・オーディエンス作成       │
└─────────────────────────────┘        └─────────────────────────────┘
        管理・配信の係                          蓄積・分析の係
```

### 比較表

| 観点 | GTM | GA4 |
|---|---|---|
| ひとことで | タグの司令塔（配信管理） | 分析の受け皿（蓄積・分析） |
| 主な役割 | タグをいつ・どう送るか管理 | 送られたデータをためて見える化 |
| 触る人 | マーケ・計測担当（コード不要） | 分析担当・マーケ |
| 出てくるID | コンテナID `GTM-XXXXXX` | 測定ID `G-XXXXXX` |
| 中心の部品 | 変数 / トリガー / タグ | プロパティ / イベント / キーイベント |
| データの保管 | しない（通すだけ） | する（ここが本体） |

### なぜ直接 gtag を貼らず、GTMを挟むのか

`gtag`（ジータグ＝GA4へ直接送るコード）をHTMLに直書きする方法もありますが、変更のたびにサイトのコードを書き換える必要があります。GTMを挟むと次の利点があります。

| 課題（直書きの場合） | GTMを挟むと |
|---|---|
| 計測を変えるたびにエンジニア依頼が必要 | GTM管理画面だけで変更・公開できる |
| タグが増えるとHTMLが散らかる | タグを一箇所で一覧管理できる |
| 「いつ送るか」をコードで書き分ける | トリガー（条件）でノーコード制御 |
| 公開前の動作確認がしづらい | プレビュー機能で事前検証できる |
| 広告タグ等も別々に貼る | GTMにまとめて配信できる |

> 注意: このデモでは学習目的で `gtag/js`（直貼りタグ）と GTM スニペットの**両方**をHTMLに置いています。実運用では二重計測を避けるため、原則どちらか一方（推奨はGTM経由）に統一します（→ 第8章）。

## 3. GA4の設定手順（受け皿を用意する）

### 原則

GA4は「アカウント → プロパティ → データストリーム」の入れ子構造です。データを受け取るには、いちばん内側の **データストリーム** を作り、そこで発行される **測定ID（`G-XXXXXX`）** をサイト（GTM）に教える必要があります。

### 図解：GA4の入れ子構造と作成の流れ

```text
GA4 アカウント (会社・組織の単位)
   └─ プロパティ (1つの分析対象 = 1サイト/アプリ群)
        └─ データストリーム (Webサイト1本ごとの入口)
                └─ 測定ID  G-XXXXXX   ← これをGTMに渡す

作成の流れ:
┌─────────┐   ┌───────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ ①アカウ │──▶│ ②プロパ   │──▶│ ③データ      │──▶│ ④測定ID      │──▶│ ⑤キーイベント│
│  ント   │   │  ティ作成 │   │  ストリーム  │   │  G-XXXXXX    │   │  (CV)を設定  │
│  作成   │   │           │   │  作成(Web)   │   │  を取得      │   │              │
└─────────┘   └───────────┘   └──────────────┘   └──────────────┘   └──────────────┘
```

### 各ステップの中身

| ステップ | やること | どこで | 補足 |
|---|---|---|---|
| ① アカウント作成 | Googleアナリティクスに会社単位の枠を作る | analytics.google.com | 既にあれば不要 |
| ② プロパティ作成 | 分析対象（このサイト）の器を作る | 管理 > プロパティを作成 | 通貨・タイムゾーンをここで設定 |
| ③ データストリーム作成 | Web用の入口を作る | データストリーム > ウェブ | サイトURLを登録 |
| ④ 測定ID取得 | `G-XXXXXX` を控える | データストリーム詳細 | これをGTMに渡す |
| ⑤ キーイベント設定 | 重要イベントをCV（コンバージョン）として登録 | 管理 > キーイベント | 例: purchase, sign_up |

> 用語: **プロパティ**＝GA4でデータをためる1単位。**データストリーム**＝そのプロパティへの入口（Web/アプリごと）。**測定ID**＝どのストリームに送るかを示す住所。**キーイベント**＝特に重要なイベント（旧称コンバージョン）。

### このデモとの接続

このデモのHTML（`index.html` ほか）には、GA4測定IDとして **プレースホルダ `G-XXXXXX`** が入っています。本番では、上の④で取得した実際の `G-XXXXXX` に置き換えます。デモで送られるイベント（`view_item` `add_to_cart` `purchase` など）のうち、`purchase` や `sign_up` を⑤でキーイベントに設定すると、GA4でCVとして集計できます。

## 4. GTMの設定手順（司令塔を組む）

### 原則

GTMは「アカウント → コンテナ」を作り、発行された **コンテナスニペット（`GTM-XXXXXX`）** をサイトに貼ります。中身は **変数・トリガー・タグ** の3部品で、これらを組み合わせて「いつ・何を・どこへ送るか」を作ります。

### 図解：GTMの3部品の関係

```text
                    ┌───────────────── GTM コンテナ (GTM-XXXXXX) ─────────────────┐
                    │                                                            │
  dataLayer ──────▶ │  ┌──────────┐    使う     ┌──────────┐    発火     ┌──────┐ │ ──▶ GA4
  event:add_to_cart │  │  変数    │ ──────────▶ │ トリガー │ ──────────▶ │ タグ │ │  (G-XXXXXX)
  ecommerce:{...}   │  │ 値の取得 │            │  いつ?   │            │ 何を?│ │
                    │  └──────────┘            └──────────┘            └──────┘ │
                    │   "値の取り出し"          "いつ送るか"          "何を送るか"│
                    └────────────────────────────────────────────────────────────┘
```

### 3部品の役割

| 部品 | 役割（ひとこと） | 例 |
|---|---|---|
| 変数 | **値の取り出し**（伝言板から値を取る） | データレイヤー変数 `ecommerce`, `user_id` |
| トリガー | **いつ送るか**（発火条件） | カスタムイベント `add_to_cart` が起きたら |
| タグ | **何を送るか**（実際の送信） | GA4イベントタグ `add_to_cart` を送る |

> 用語: **データレイヤー変数**＝dataLayerに書かれた値（`ecommerce` や `user_id`）をGTMで取り出す箱。**カスタムイベント トリガー**＝dataLayerの `event` 名（`add_to_cart` など）が来たら発火する条件。

### 設定の流れ

```text
┌──────────┐  ┌──────────┐  ┌──────────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐
│①アカウン │─▶│②コンテナ │─▶│③スニペット   │─▶│④変数を   │─▶│⑤トリガー │─▶│⑥タグを設定   │
│ ト作成   │  │  作成    │  │  GTM-XXXXXX  │  │  作成    │  │  を作成  │  │ (設定+イベント)│
│          │  │ (Web)    │  │  をサイトへ  │  │          │  │          │  │              │
└──────────┘  └──────────┘  └──────────────┘  └──────────┘  └──────────┘  └──────────────┘
```

### GA4タグは2種類ある（重要）

GTMからGA4へ送るとき、タグは大きく2種類に分けて考えます。

| タグ | 役割 | いつ発火 | 送る内容 |
|---|---|---|---|
| GA4設定タグ（Googleタグ） | GA4との接続を初期化し、`page_view` を担う土台 | 全ページ（初期化トリガー） | 測定ID `G-XXXXXX`、共通設定、ユーザープロパティ |
| GA4イベントタグ | 個別イベントを送る | 該当イベント発生時（カスタムイベント トリガー） | `add_to_cart` などのイベント名＋パラメータ・eコマースデータ |

```text
全ページ                      特定の操作のとき
┌──────────────┐             ┌──────────────┐
│ GA4設定タグ  │             │ GA4イベント  │
│ (Googleタグ) │  土台の上に │   タグ       │
│ G-XXXXXX     │ ◀───────── │ add_to_cart  │
│ page_view    │  載る形     │ view_item ...│
└──────────────┘             └──────────────┘
```

### このデモとの接続

デモHTMLにはGTMスニペット **`GTM-XXXXXX`**（プレースホルダ）が入っています。本番では実コンテナIDに置き換え、GTM側で次を作ります。

- **変数**: データレイヤー変数 `ecommerce`、`user_id`、`user_properties`
- **トリガー**: カスタムイベント `view_item` `add_to_cart` `purchase` など
- **タグ**: GA4設定タグ1つ ＋ 各イベント用のGA4イベントタグ

## 5. eコマース計測の考え方（商品データを送る）

### 原則

eコマース計測では、ただイベント名を送るだけでなく **「どの商品が・いくつ・いくらで」** という商品の明細を一緒に送ります。その明細を入れる箱が `ecommerce` の中の **`items[]`（商品の配列）** です。

### 図解：eコマースデータの流れ

```text
┌──────────────────────────┐   ┌──────────────────────┐   ┌──────────────────────┐
│ サイト                   │   │ GTM                  │   │ GA4                  │
│ dataLayer.push({         │   │ GA4イベントタグで    │   │ レポート「eコマース」│
│   event:'add_to_cart',   │──▶│ 「eコマースデータを  │──▶│ で view_item /       │
│   ecommerce:{            │   │  使用」をON          │   │ add_to_cart /        │
│     items:[ {...} ]      │   │ → items[] を読む     │   │ purchase 等が見える  │
│   }})                    │   │                      │   │                      │
└──────────────────────────┘   └──────────────────────┘   └──────────────────────┘
```

### items[] の主要フィールド

| フィールド | 意味 | このデモでの例 |
|---|---|---|
| `item_id` | 商品ID（品番） | `351001` |
| `item_name` | 商品名 | エアリズムコットン オーバーサイズT（半袖） |
| `item_brand` | ブランド | UNILO |
| `item_category` | カテゴリ大分類 | ウィメンズ |
| `item_category2` | カテゴリ中分類 | トップス |
| `price` | 単価（税込） | `1500` |
| `quantity` | 数量 | `2` |
| `item_variant` | バリエーション（色/サイズ） | ホワイト |
| `index` | 一覧での並び順 | `0` |
| `item_list_id` / `item_list_name` | どの一覧から来たか | `category_women` / WOMEN 一覧 |

### 主なeコマースイベント（GA4標準）

| イベント | 意味 | デモで発火する場所 |
|---|---|---|
| `view_item_list` | 一覧の表示（インプレッション） | 一覧ページ（list.html）／トップ（index.html のトップ おすすめグリッド） |
| `select_item` | 一覧から商品クリック | 一覧ページのカード／トップ（index.html のトップ おすすめグリッド） |
| `view_item` | 商品詳細の表示 | 商品ページ（product.html） |
| `add_to_cart` | カートに追加 | 商品ページ |
| `remove_from_cart` | カートから削除 | カートページ（cart.html） |
| `view_cart` | カート表示 | カートページ |
| `begin_checkout` | 購入手続き開始 | チェックアウト（checkout.html） |
| `add_shipping_info` | 配送情報入力 | チェックアウト |
| `add_payment_info` | 支払い情報入力 | チェックアウト |
| `purchase` | 購入完了 | 完了ページ（complete.html） |

### このデモとの接続

`assets/js/datalayer.js` の `toItem()` が上の `items[]` を組み立て、各メソッド（`viewItem` `addToCart` `purchase` など）が `ecommerce` 付きでpushします。重要なのは、毎回 **`ecommerce: null` でリセットしてから** pushしている点です（前のイベントの `items` が次に混ざるのを防ぐGA4推奨パターン → 第8章で詳述）。

```text
// datalayer.js の ecom() がやっていること
push({ ecommerce: null });                       // ① まず空にする（リセット）
push({ event:'add_to_cart', ecommerce:{ ... } }); // ② その上で本物を送る
```

## 6. ユーザープロパティと user_id

### 原則

**ユーザープロパティ** は「そのユーザーがどんな人か」を表す属性（会員ランク・ログイン状態など）で、**user_id** は「同じ人だ」と複数端末・複数訪問をまたいで識別するためのIDです。これらを送ると、GA4で「会員ランク別の購入率」のような分析ができます。

### 図解：user_id / ユーザープロパティの流れ

```text
┌───────────────────────────┐   ┌──────────────────────────┐   ┌──────────────────────────┐
│ サイト (会員DB由来)       │   │ GTM                      │   │ GA4                      │
│ dataLayer.push({          │   │ 変数でマッピング:        │   │ ・user_id でクロス端末   │
│   user_id:'U123',         │──▶│  user_id → User-ID       │──▶│  識別                    │
│   user_properties:{       │   │  user_properties →       │   │ ・ユーザープロパティで   │
│     login_state,          │   │   ユーザープロパティ     │   │  会員ランク別分析        │
│     membership_rank }})    │   │ → GA4設定タグに載せる    │   │                          │
└───────────────────────────┘   └──────────────────────────┘   └──────────────────────────┘
```

### 何のために使うか

| 使いみち | 例 |
|---|---|
| 会員ランク別の分析 | ゴールド会員 vs 一般会員の購入率比較 |
| ログイン状態別の分析 | ログイン済 vs ゲストの行動差 |
| クロス端末の同一人物把握 | スマホで見て、PCで買った人を1人として数える |

### dataLayer → GTM → GA4 のマッピング

| dataLayerのキー | GTM変数 | GA4での扱い |
|---|---|---|
| `user_id` | データレイヤー変数 `user_id` | User-ID（ユーザー識別） |
| `user_properties.login_state` | データレイヤー変数 | ユーザープロパティ `login_state` |
| `user_properties.membership_rank` | データレイヤー変数 | ユーザープロパティ `membership_rank` |

> 注意: GA4のユーザープロパティは事前に「管理 > カスタム定義」で登録する必要があります。また、個人を直接特定できる情報（氏名・メール等）はそのまま送らないのが原則です。

### このデモとの接続

`datalayer.js` の `setUser()` が、ログイン状態に応じて次をpushします。

```text
setUser(user) → push({
  user_id: ログイン時は user_id、ゲスト/ログアウト時は null,
  user_properties: {
    login_state:    'logged_in' or 'guest',
    membership_rank: 会員ランク or 'none'
  }
})
```

この `setUser()` は `common.js` から **全ページで** 呼ばれ、ログイン/会員登録時（`login.html`）にも呼ばれます。`mypage.html` には「会員ID(user_id)と会員ランクは全ページでGA4のユーザープロパティとして送信される」と注記もあります。本番ではこれらの値は会員DBがサーバー側で確定してHTMLに埋め込みます（デモでは `store.js` が代役）。

## 7. このデモサイトでの確認方法

### 原則

計測は「送ったつもり」では不十分で、**実際にデータが出ているか** を3段階で確認します。① ブラウザのコンソールでdataLayerを見る → ② GTMプレビューでタグ発火を見る → ③ GA4 DebugViewで着地を見る、の順です。

### 図解：3段階の確認

```text
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐
│ ① コンソール     │   │ ② GTMプレビュー  │   │ ③ GA4 DebugView  │
│ dataLayerを確認  │──▶│ タグ発火を確認   │──▶│ GA4着地を確認    │
│ 「書けてる?」    │   │ 「拾えてる?」    │   │ 「届いてる?」    │
└──────────────────┘   └──────────────────┘   └──────────────────┘
   サイト側の出力          GTMが拾ったか           GA4に入ったか
```

### 起動手順（ローカルサーバー）

`file://` で直接開くと一部が動かないため、簡易サーバーで配信します。

```text
# プロジェクト直下で実行
python3 -m http.server 8000

# ブラウザで開く
http://localhost:8000/index.html
```

### ① コンソールでdataLayerを確認

このデモは各HTMLの先頭で `window.__DL_DEBUG__ = true;` を設定しているため、pushされた内容が自動でコンソールに `[dataLayer] {...}` と出力されます。

```text
ブラウザの「検証(DevTools)」→ Console を開く
  → 操作するたびに [dataLayer] { event:'add_to_cart', ecommerce:{...} } が出る
  → window.dataLayer と入力すれば、たまった配列を全部見られる
```

### ② GTMプレビュー

GTMの管理画面「プレビュー」→ Tag Assistant で `http://localhost:8000/...` を開き、操作ごとに左の Summary でイベントを選ぶと、そのイベントで **どのタグが発火したか / 変数に何が入ったか** を確認できます。

### ③ GA4 DebugView

GTMプレビュー中（またはGA4のデバッグ有効時）、GA4管理画面の「管理 > DebugView」に、ほぼリアルタイムでイベントが流れます。`add_to_cart` をクリックすると、`items` パラメータの中身まで確認できます。

> 用語: **DebugView**＝GA4で開発中のデータをリアルタイムに見る検証用画面。本番レポートには影響しません。

### どのページ操作でどのイベントが出るか（早見表）

## 8. つまずきやすいポイント

| つまずき | 何が起きる | 原因 | 対処 |
|---|---|---|---|
| 二重計測 | 1回の操作が2回数えられる | `gtag`直貼りとGTM経由のGA4タグが両方動いている | GA4送信は片方に統一（推奨はGTM経由）。このデモは学習用に両方置いてある点に注意 |
| ecommerce が残る | 前の商品のitemsが次のイベントに混ざる | `ecommerce` をリセットせず連続pushした | 各pushの前に `push({ecommerce:null})` を入れる（このデモの `ecom()` が実施済み） |
| page_view の重複 | ページビューが二重に出る | GA4の拡張計測「ページビュー」自動オン＋手動page_viewタグも設置 | どちらか一方に。基本は自動計測に任せる |
| イベントが届かない | DebugViewに何も出ない | トリガー名とdataLayerの`event`名が不一致 | 文字列を完全一致で確認（例: `add_to_cart`） |
| 変数が空 | itemsやuser_idが空で送られる | データレイヤー変数のキー名ミス／pushより先にタグ発火 | 変数のキー名を確認。pushの順序（値→イベント）を守る |
| user_id が出ない | クロス端末識別ができない | ゲスト時にも送ろうとした／未ログイン | ログイン確定後のみ送る（デモは `logged_in` 時のみ送信） |
| 値が文字列/数値で不一致 | 集計がずれる | priceやquantityを文字列で送った | 数値は数値型で送る（デモは数値で送信） |
| ローカルで動かない | スクリプトが読めない | `file://` で直接開いた | `python3 -m http.server` 経由で開く |

> 二重計測の補足: このデモは「直貼りgtag」と「GTM」の両方の置き方を学べるよう、あえて両方をHTMLに残しています。本番に持っていく際は、GA4へのイベント送信元を **どちらか一方に必ず統一** してください。

## まとめ（1枚）

```text
┌──────────────────────────────────────────────────────────────────────┐
│  サイト操作 → DL.xxx() → dataLayer.push → GTM(変数/トリガー/タグ) → GA4 │
│                                                                        │
│   dataLayer = 伝言板     GTM = 司令塔(GTM-XXXXXX)   GA4 = 受け皿(G-XXXXXX) │
│   ・eコマースは items[] で商品明細を一緒に送る                          │
│   ・ユーザー属性は user_id / user_properties で送る                     │
│   ・確認は コンソール → GTMプレビュー → GA4 DebugView の3段階           │
└──────────────────────────────────────────────────────────────────────┘
```

このデモで「サイトがdataLayerに何を書くか」を `assets/js/datalayer.js` で読み、「GTMがそれをどう拾ってGA4へ渡すか」をGTMプレビューで見れば、計測の全体像が手を動かしながら理解できます。
